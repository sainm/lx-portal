# AI 聊天后续集成文档

## 目标

AI 聊天能力用于在用户浏览心理咨询门户网站时提供低门槛的信息引导，包括服务介绍、预约前问题澄清、常见咨询流程说明和文章内容导览。

该能力不替代心理咨询师、不提供医学诊断、不提供急救服务，也不应承诺疗效。涉及自伤、自杀、伤人、被伤害等高风险内容时，应优先触发危机提示和人工介入流程。

## 当前预留接口设计

当前项目尚未实现 AI 聊天接口。后续建议沿用现有公共接口风格：

- 公共接口统一放在 `/api/public` 下。
- 响应结构沿用 `ApiResponse<T>`：`code`、`message`、`data`。
- 请求参数使用 DTO/record 并配合 Jakarta Validation 做基础校验。
- 业务编排放在 service 层，controller 只负责 HTTP 入参、鉴权边界和响应封装。
- 关键用户行为可复用 `/api/public/track` 的埋点思路记录事件，但避免保存敏感对话全文。

建议预留以下接口：

```http
POST /api/public/ai-chat/sessions
Content-Type: application/json
```

创建匿名聊天会话。可选参数包括页面来源、入口位置、用户当前关注方向等。

```json
{
  "pagePath": "/",
  "entry": "home_chat_widget",
  "concernDirection": "压力"
}
```

建议响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "sessionId": "uuid",
    "welcomeMessage": "你好，我可以帮你了解咨询流程和预约方式。若你正处于紧急危险中，请立即联系当地急救、警方或可信任的人。"
  }
}
```

```http
POST /api/public/ai-chat/messages
Content-Type: application/json
```

发送用户消息并返回 AI 回复。

```json
{
  "sessionId": "uuid",
  "message": "我最近压力很大，不知道适合预约哪类咨询",
  "pagePath": "/services"
}
```

建议响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "reply": "听起来你正在经历持续压力。我可以先帮你了解适合的咨询方向，但这里不能替代正式评估。你可以考虑从情绪压力或职场压力方向开始预约。",
    "crisisDetected": false,
    "handoffRecommended": false,
    "suggestedActions": [
      {
        "type": "appointment",
        "label": "预约首次咨询",
        "url": "/appointment"
      }
    ]
  }
}
```

```http
POST /api/public/ai-chat/feedback
Content-Type: application/json
```

记录用户对单条回复的反馈。只记录必要字段，例如 `sessionId`、`messageId`、`rating`、`reason`，不要记录额外隐私信息。

## 后续接入 AI provider 的步骤

1. 新增配置项

   在 `application.yml` 中增加 `ai.chat` 配置段，使用环境变量注入密钥、模型名、超时、开关和限流参数。开发、测试、生产环境分别配置默认值。

2. 定义 provider 抽象

   建议新增 `AiChatProvider` 接口，屏蔽不同厂商 SDK 或 HTTP API 差异。service 层只依赖内部接口，避免 controller 直接调用外部 provider。

3. 实现请求前置处理

   在调用 provider 前完成输入长度限制、空白内容校验、危机关键词检测、敏感信息脱敏、频率限制和会话状态检查。

4. 设计系统提示词

   系统提示词应固定平台边界：

   - 只做心理咨询前的信息引导和情绪支持。
   - 不做诊断、不开药、不判断病情严重程度。
   - 不承诺治疗效果。
   - 遇到危机内容时停止普通聊天，返回危机提示。
   - 鼓励用户预约专业咨询师或联系当地紧急资源。

5. 增加人工转接策略

   当用户表达强烈痛苦、持续危险、未成年人风险、家暴或伤人风险时，AI 回复应建议联系人工工作人员，并引导用户使用预约表单或官方联系方式。

6. 接入日志与审计

   记录调用时间、provider、模型、耗时、token 用量、是否触发危机、是否失败等运维字段。对话正文默认不落库；如业务确需保存，应先完成隐私告知、保留期限和访问权限设计。

7. 增加降级方案

   provider 超时、限流或不可用时，返回固定兜底回复，并提供预约入口、服务介绍入口和危机提示。

8. 上线灰度

   先通过配置开关仅对测试环境或少量入口开放，观察失败率、危机触发率、用户反馈和预约转化，再逐步扩大范围。

## 环境变量

建议使用以下环境变量，具体名称可在实现时按项目配置规范调整：

| 变量名 | 说明 | 示例 |
| --- | --- | --- |
| `AI_CHAT_ENABLED` | 是否启用 AI 聊天 | `false` |
| `AI_PROVIDER` | provider 标识 | `openai`、`azure_openai`、`qwen` |
| `AI_API_KEY` | provider API key | 不应提交到仓库 |
| `AI_BASE_URL` | provider API base URL | `https://api.example.com/v1` |
| `AI_MODEL` | 聊天模型名称 | `gpt-4.1-mini` |
| `AI_REQUEST_TIMEOUT_MS` | 单次请求超时 | `10000` |
| `AI_MAX_INPUT_CHARS` | 用户单条消息最大长度 | `1000` |
| `AI_MAX_OUTPUT_TOKENS` | 回复最大 token 数 | `500` |
| `AI_DAILY_LIMIT_PER_IP` | 单 IP 每日调用上限 | `50` |
| `AI_STORE_CONVERSATION` | 是否保存对话正文 | `false` |
| `AI_REDACTION_ENABLED` | 是否启用敏感信息脱敏 | `true` |
| `AI_CRISIS_KEYWORDS` | 危机关键词配置，建议支持逗号分隔或配置文件 | `自杀,自伤,想死` |

生产环境要求：

- `AI_API_KEY` 只能通过环境变量、密钥管理服务或部署平台 secret 注入。
- 不得写入 `application-prod.yml`、README 示例真实值或日志。
- 启用请求超时、重试上限和限流，避免 provider 异常拖垮站点。

## 边界与合规提示

页面、接口和 AI 回复中都应保持清晰边界：

- AI 仅提供一般性信息、情绪支持和预约引导。
- AI 不构成医疗、心理诊断、治疗建议或法律建议。
- AI 不能替代持证心理咨询师、心理治疗师、精神科医生或急救人员。
- 用户不应在聊天中输入身份证号、详细住址、病历、银行卡、账号密码等敏感信息。
- 对未成年人相关内容应更谨慎，必要时建议监护人参与并联系专业机构。
- 对家暴、性侵、虐待、伤人风险等内容，应提示用户优先确保人身安全并联系当地紧急资源。
- 对话数据如需保存，必须提供隐私告知、用途说明、保存期限、删除方式和后台访问控制。
- 运营人员查看任何聊天记录前，应具备明确权限、审计记录和最小化访问原则。

建议在聊天入口附近展示短提示：

```text
AI 助手仅用于咨询前信息引导，不能替代专业咨询、诊断或急救服务。若你或他人正处于立即危险中，请马上联系当地急救、警方或可信任的人。
```

## 危机关键词处理

危机处理应在调用 AI provider 前完成第一层检测，调用后再对回复做第二层安全检查。

### 关键词分类

高危关键词示例：

- 自杀、自尽、想死、不想活、结束生命
- 自伤、割腕、吞药、跳楼、上吊
- 杀人、伤人、报复、带刀、放火
- 被家暴、被虐待、被性侵、被控制
- 现在就要、已经准备好、写遗书、买了药

中危表达示例：

- 活着没意思、撑不下去、没有希望
- 很想消失、没人会在乎、太痛苦了
- 控制不住自己、怕自己做傻事

### 处理策略

命中高危关键词时：

- 不再进入普通 AI 问答流程。
- 返回固定危机提示文案。
- 建议用户立即联系当地急救电话、警方、可信任的人或线下医疗机构。
- 如平台有人工值班能力，标记 `handoffRecommended=true` 并通知工作人员。
- 记录危机事件元数据，例如时间、入口、关键词类型、sessionId，但默认不记录完整正文。

命中中危表达时：

- 可以返回更谨慎的支持性回复。
- 鼓励用户尽快联系可信任的人或专业机构。
- 引导预约咨询，但不能把预约作为紧急情况的唯一建议。
- 可建议用户如果有立即危险，马上联系急救或警方。

建议高危固定回复：

```text
我很抱歉你正在经历这么强烈的痛苦。这里不能提供急救服务。如果你或他人正处于立即危险中，请立刻联系当地急救电话、警方，或马上联系身边可信任的人陪在你身边。若可以，请先远离可能伤害自己的物品，并到有人陪伴的安全地点。
```

### 技术注意事项

- 关键词表应支持配置化，便于运营和专业人员调整。
- 检测应忽略空格、标点和常见谐音变体。
- 不要仅依赖关键词；后续可增加 provider moderation 或本地分类模型。
- 危机提示不应包含具体自伤方法、工具、剂量或操作步骤。
- 危机事件通知应避免在企业微信、飞书等渠道暴露完整敏感内容。

## 测试清单

### 单元测试

- AI 聊天开关关闭时，接口返回明确的不可用提示。
- 请求消息为空、超长、缺少 sessionId 时返回校验错误。
- provider 超时、限流、认证失败时返回降级回复。
- 高危关键词命中时不调用 provider。
- 中危表达命中时返回谨慎提示并标记风险等级。
- 回复结果包含预约入口时，URL 指向 `/appointment`。
- 日志中不包含 `AI_API_KEY` 和完整敏感对话正文。

### 集成测试

- `POST /api/public/ai-chat/sessions` 能创建会话并返回 `sessionId`。
- `POST /api/public/ai-chat/messages` 能按 `ApiResponse` 格式返回 `reply`。
- 限流达到阈值后返回友好提示。
- provider mock 返回异常内容时，服务端安全检查能拦截或替换。
- 危机事件能写入审计或通知队列，但不暴露过量隐私。

### 前端测试

- 聊天入口展示边界提示。
- 首次打开聊天窗口展示欢迎语和紧急情况提示。
- 网络失败、provider 不可用、限流时有可理解的错误文案。
- 高危输入后界面突出显示危机提示和紧急建议。
- 移动端聊天窗口不遮挡预约按钮、隐私提示和表单提交按钮。

### 安全与合规测试

- API key 不出现在前端源码、接口响应、日志和错误栈中。
- CORS、CSRF、频率限制和输入长度限制符合公共接口要求。
- 敏感字段脱敏策略有效，例如手机号、身份证号、详细住址。
- 后台访问聊天相关数据时有权限校验和操作审计。
- 数据保留期限、删除策略和隐私政策文案已确认。

### 人工验收

- 专业人员审阅系统提示词、危机提示文案和边界提示。
- 运营人员确认人工转接流程和通知渠道。
- 技术负责人确认 provider SLA、费用预算和降级方案。
- 上线前在测试环境完成至少一轮高危、中危、普通咨询、无关问题和恶意输入场景演练。
